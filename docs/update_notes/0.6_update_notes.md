# 0.6 版本更新日志

## 变更概览

本次更新主要包含：**传感器 / 设备在线状态显示修复**（admin 列表与详情页打勾图标"永远在线"问题）、**Admin 一致性整治**（自定义在线过滤器、`is_online` / `last_seen` 只读化、消除 N+1 查询）、**卡片拖拽排序**（前端 vuedraggable + 后端 `sort_order` 字段 + reorder API）、`update_heartbeat` 接收数据时间戳。

---

## 一、在线状态显示修复（核心 Bug）

### 背景

`devices` admin 列表里"在线状态"图标永远显示 ✓，即使设备已离线数小时。详情页"状态信息"里的 `is_online` BooleanField 也是同样的打勾，造成误判。

根因：`Device.is_online` 字段只在收到心跳时被置为 `True`，**只有手动触发 `check_online_status()` action 才会被回写为 `False`**——平台没有定时任务、信号或中间件去周期性扫描，所以一旦上报过，字段永远为真。

`sensors` 那边已通过 `Sensor.computed_is_online` 计算属性（3 分钟阈值）规避了这个问题，但 `devices` 没有对应实现。

### 变更

- **新增** `Device.computed_is_online` 计算属性：基于 `last_seen` 与 `heartbeat_interval × 3`（默认 60s × 3 = 180s）实时判断
- **修改** `devices/admin.py:online_indicator`：改用 `computed_is_online`，离线时显示"● 离线 (X 分钟/小时/天前 上报)"，与 `sensors/admin.py:online_status` 视觉对齐
- **移除** `devices` 详情页 `fieldsets` 状态信息分组中的 `is_online` 字段（避免那个会误导人的打勾图标），与 `sensors` 详情页保持一致；`is_online` 字段本身保留在数据库中

### 文件变更

| 文件                  | 变更                                  |
|-----------------------|---------------------------------------|
| `devices/models.py`   | 新增 `computed_is_online` property    |
| `devices/admin.py`    | `online_indicator` 改实时计算；移除 fieldsets 中的 `is_online` |

## 二、Admin 一致性整治

### 背景

`sensors` 与 `devices` 的 admin 在多个细节上不一致或存在隐患：`list_filter` 用裸 `is_online` 字段（与列表显示的实时状态不一致）、`last_seen` 可编辑（容易被人为弄脏）、`latest_data_time` 每行触发一次 `data_records.first()` 的 N+1 查询。

### 变更

- **新增** `SensorOnlineFilter` / `DeviceOnlineFilter`（`SimpleListFilter` 子类）：基于 `last_seen` 的实时窗口过滤，替换原有的 `is_online` 字段过滤
- **`is_online` / `last_seen` 加入 `readonly_fields`**：系统自动维护字段不再可手动编辑
- **优化 `latest_data_time`**：优先读 `obj.last_seen`，仅在为空时回退查 `data_records.first()`，列表页消除 N+1
- **清理代码噪音**：
  - 移除 `SensorTypeAdmin` 中无效的 `filter_horizontal = []`
  - 删除 `SensorDataAdmin` / `SensorStatusCollectionAdmin` 中两处局部重复 `import json`
  - `except:` 改为具体异常 `except (TypeError, ValueError)`

### 文件变更

| 文件                  | 变更                                                 |
|-----------------------|------------------------------------------------------|
| `sensors/admin.py`    | `SensorOnlineFilter`、readonly、N+1 优化、清理       |
| `devices/admin.py`    | `DeviceOnlineFilter`、readonly、N+1 优化             |

## 三、`update_heartbeat` 接收数据时间戳

### 背景

`Device.update_heartbeat()` 在被 `DeviceData.save()` 调用时一律使用 `timezone.now()` 写入 `last_seen`，与 `Sensor.update_last_seen(self.timestamp)` 行为不一致——历史数据回灌时，`last_seen` 会被覆盖成"现在"，与数据本身的时间戳脱钩。

### 变更

- **修改** `Device.update_heartbeat(timestamp=None)`：新增可选 `timestamp` 参数，缺省回退 `timezone.now()`，向后兼容现有无参调用
- **修改** `DeviceData.save()`：调用时传入 `self.timestamp`，与 `SensorData.save()` 行为对齐

### 文件变更

| 文件                  | 变更                              |
|-----------------------|-----------------------------------|
| `devices/models.py`   | `update_heartbeat` 接收 timestamp |

## 四、卡片拖拽排序

### 背景

设备 / 传感器管理界面采用卡片网格展示，过去仅按 `-created_at` 单一排序，无法按用户使用偏好调整显示顺序。

### 变更

#### 后端

- **新增** `Sensor.sort_order` / `Device.sort_order` 字段（`IntegerField(default=0, db_index=True)`）
- **修改** Meta `ordering` 为 `['sort_order', '-created_at']`：sort_order 升序优先，相同时按创建时间倒序（保留"最新创建在前"的二级语义）
- **新增** `POST /sensors/reorder/` 与 `POST /devices/reorder/`：请求体 `{"order": ["id_a", "id_b", ...]}`，事务内按数组顺序写入 `sort_order = 1, 2, ...`，仅 `IsAdminUser` 可调用
- **数据迁移**：`0004_sensor_sort_order.py` / `0009_device_sort_order.py` 通过 `RunPython` 给已有记录按 `-created_at` 顺序赋值 `sort_order = 1, 2, ...`，**保持当前显示顺序不变**
- **序列化器**：列表 serializer 暴露 `sort_order` 只读字段

#### 前端

- **新增依赖** `vuedraggable@^4.1.0`（Vue 3 兼容版，底层为 sortablejs）
- **新增 API** `reorderSensors(order)` / `reorderDevices(order)`
- **改造视图** `SensorsView.vue` / `DevicesView.vue`：卡片网格替换为 `<draggable v-model="...">`，拖拽结束 `@end` 触发 reorder 请求，失败时回滚刷新
- **拖拽禁用条件**：非 staff 用户、或筛选/搜索激活时（`hasActiveFilter`）禁用拖拽。原因：reorder 接口将"当前可见列表"按顺序写为 `sort_order = 1..N`，筛选状态下被筛掉的项 `sort_order` 会与可见项冲突
- **视觉反馈**：新增 `.drag-ghost` 样式（虚线珊瑚橙边框 + 半透明），与 0.5 引入的 Claude 风格主题色对齐

### 设计要点

- **新建项默认 `sort_order = 0`**：会排在所有已拖拽过的项前面，对应"最新创建在最前"的语义
- **批量重写而非增量更新**：每次拖拽提交完整可见列表的新顺序，简单可靠，无并发竞争
- **后端事务保护**：`reorder` 用 `transaction.atomic()` 包裹，失败时整体回滚

## 五、升级步骤

### Docker 部署

```bash
# 拉新代码后，一次性重新 build 后端 + 前端镜像
docker compose down
docker compose up -d --build
# backend 启动会自动跑 migrate --noinput，
# 应用 0004_sensor_sort_order 与 0009_device_sort_order
# data migration 会按 -created_at 自动给现有记录赋 sort_order，保持原有显示顺序
```

测试环境（源码挂载）：

```bash
docker compose -f docker-compose.test.yml down
docker compose -f docker-compose.test.yml up -d
# 前端因为 package.json 有变更，需要触发 npm install：
docker compose -f docker-compose.test.yml exec frontend-test npm install
docker compose -f docker-compose.test.yml restart frontend-test
```

### 本地部署

```bash
git pull
cd iot_control_platform
python manage.py migrate sensors devices
# 重启 runserver / gunicorn

cd ../frontend
npm install        # 安装 vuedraggable
npm run build      # 或 npm run dev
```

### 注意事项

- **`package-lock.json` 同步更新**：本次新增 `vuedraggable@4.1.0` 与 `sortablejs@1.14.0` 两个依赖；如果用 `npm ci` 构建（包括 `frontend/Dockerfile` 中的步骤），必须先在本地执行 `npm install --package-lock-only` 把 lock 文件更新提交，否则 `npm ci` 会因 `Missing: vuedraggable from lock file` 失败
- **数据迁移幂等性**：`init_sort_order` 在重复执行时会再次写入 `sort_order`，但仅在首次部署到 0.6 时有意义；`RunPython` 的 `reverse_code=noop` 表示回滚迁移不会清空 `sort_order`

## 六、设计要点速查

1. **在线状态判断**统一为基于 `last_seen` 的实时计算
   - sensors：`computed_is_online` 用硬编码 3 分钟阈值
   - devices：`computed_is_online` 用 `heartbeat_interval × 3`（默认也是 3 分钟）
2. **`is_online` 数据库字段保留**，外部 API（如 `plugins/data_viz/views.py:84`）继续使用；admin 内部一律改用 `computed_is_online`
3. **拖拽排序在筛选状态下禁用**，避免被筛掉的项 `sort_order` 与可见项冲突
4. **`update_heartbeat(timestamp=None)`** 与 `update_last_seen(timestamp=None)` 接口对齐，两边都使用数据自带的 timestamp
5. **批量 reorder 而非增量**：前端提交完整列表新顺序，后端事务内重写 `sort_order = 1..N`

---

## 七、已知限制 / 后续可优化

- **`plugins/data_viz/views.py:84` 仍直接读 `d.is_online` 字段**：API 返回的 `is_online` 不会随时间衰减为离线。如该 API 被前端用于"在线指示灯"展示，建议改用 `d.computed_is_online`
- **筛选状态下无法拖拽**：UX 上是合理保护，但若用户期望"在筛选视图内排序"，需要后端支持"按筛选范围重写 `sort_order`"的更复杂语义，目前不实现
- **新建项默认 `sort_order = 0`**：会排在所有已拖拽项前面。如果你的工作流是"新建后想固定到末尾"，可改为在 `Sensor.save()` / `Device.save()` 中动态取 `max(sort_order) + 1`，但需考虑并发写入的一致性
- **拖拽手柄是整张卡片**：`handle=".sensor-drag-handle, .iot-card"` 让全卡片可拖。如果出现"想点开却拖出去了"的场景，可以加专门的拖拽手柄元素（如卡片右上角 ⋮⋮ 图标）并把 handle 改为只认手柄
