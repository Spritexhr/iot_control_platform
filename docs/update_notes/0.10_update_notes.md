# 0.10 版本更新日志

> 发布版本：`v0.10.0`  
> 发布日期：2026-06-28  
> 变更范围：`f9f99af`（0.9 自动化引擎重构）至 `v0.10.0`

## 版本概览

0.10 的主线是把面向单一装置的 EB 大屏、P&ID 画板和时序查询能力，统一提升为平台原生 **Project 项目/场景系统**。一个 Project 可以按房间或工段组织传感器、设备、视图和控制方案，不再向通用 Sensor/Device 模型写入插件字段。

本版本同时加入双位、PI、PID 结构化闭环控制，扩展仿真管理和工业传感器类型，并完成前端与文档体系整理。

主要亮点：

- 原生 Project / Section / Member / View 数据模型
- 卡片、P&ID、时序趋势、自动化控制四类场景视图
- Project 专属 WebSocket 实时通道
- 双位、PI、PID 控制方案与单实例调度
- 仿真 Web 管理界面和通用节点注册机制
- 工业温度、压力、流量、液位传感器
- Docker 启动顺序和健康检查加固
- 旧 EB/P&ID 插件彻底移除，文档归并为平台级设计与指南

## 一、Project 项目/场景系统

### 原生数据模型

新增 `projects` Django app：

- `Project`：工业装置、智能家居或自定义场景
- `ProjectSection`：项目内的房间、工段或区域
- `ProjectSensorMember`：场景传感器点位及位号、单位、阈值、严重度
- `ProjectDeviceMember`：场景设备及位号、区域、可见性
- `ProjectView`：卡片、流程图、趋势和控制视图

主模型保持通用：Sensor/Device 不包含 `plant_code`、`plant_metadata` 或具体插件字段。同一资源可以加入多个 Project 或 Section，并保存不同的场景元数据。

### 房间级隔离

成员和视图必须属于 Section：

- 传感器唯一约束：`(section, sensor, data_key)`
- 设备唯一约束：`(section, device)`
- 多字段传感器按 `sensor_id::data_key` 拆分点位
- 同一资源可以在不同房间复用
- 视图只展示所属房间成员

### REST 与 WebSocket

新增主要接口：

```text
/api/projects/
/api/projects/<id>/layout/
/api/projects/<id>/snapshot/
/api/projects/<id>/series/
/api/projects/<id>/bindable_sources/
/api/project_sections/
/api/project_sensor_members/
/api/project_device_members/
/api/project_views/
```

实时端点：

```text
/ws/projects/<project_id>/?token=<jwt>
```

工作台先通过 REST 获取 layout/snapshot，再通过 WebSocket 接收 `sample` 和 `device.status` 增量。

## 二、四类场景视图

### 卡片大屏

- 传感器实时值、单位、更新时间和阈值状态
- 设备在线状态与命令面板
- 搜索、在线和报警过滤
- 离线遮罩、迷你趋势线和阈值提示

### P&ID / 流程图

- Vue Flow 编辑和运行模式
- 集中式 `symbols.js` 图元注册表
- 节点旋转、镜像和动态属性
- 仪表按 `point_id` 绑定实时值
- 物流入口/出口、容器等工业图元
- 独立 Symbol Editor 辅助工具

原 `plant_diagram` 插件能力已迁入 `frontend/src/views/projects/diagram/`。

### 时序趋势

- ECharts 多字段曲线
- 1 小时、6 小时、24 小时、7 天及自定义时间窗
- 传感器采集值和设备状态事件
- 心跳事件过滤
- 图表容器 ResizeObserver 自适应

### 自动化控制

- 双位控制：带回差/死区
- PI：比例积分、限幅和抗积分饱和
- PID：增加微分项
- heat/cool 作用方向
- 模拟量或开关量命令映射
- 单拍测试、启用、停用和运行状态展示

## 三、ControlScheme 结构化控制

新增 `automation.ControlScheme`、控制算法和 API：

```text
ProjectSensorMember → PV
ProjectDeviceMember → 执行器
SP + 算法参数 → 输出 → Device 命令
```

生产调度器只在单实例 `iot-mqtt-runner` 中运行；多 ASGI worker 的 backend 不执行控制环，避免重复下发命令。

权限：

- 已登录用户：查看
- 工作人员：启用、停用、单拍
- 超级用户：创建、修改、删除

## 八、不兼容变化与升级提示

### 旧插件入口移除

以下入口不再存在：

```text
/plugins/eb_plant
/plugins/plant-diagram
/api/plugins/eb_plant/
/api/plugins/plant_diagram/
```

应在 `/projects` 中创建 Project、Section 和对应视图。

### 清理旧插件登记

升级后执行：

```bash
docker exec iot-backend python manage.py sync_plugins --prune
```

该命令会移除数据库中已不存在的 `eb_plant` / `plant_diagram` 插件登记，仅保留文件系统中实际存在的插件。

### 删除保护

ControlScheme 对 Sensor/Device member 使用 `PROTECT`。存在控制方案时，删除 Project、Section、Sensor、Device 或成员都会被阻止；请先停用并删除相关控制方案。

### 默认视图和 Section 一致性

- `ProjectView.is_default` 当前没有数据库唯一约束，调用方应避免同一房间存在多个默认视图。
- ControlScheme API 校验同 Project，但尚未在数据库强制两个 member 与 Section 完全一致；建议始终从 Project 工作台配置。

## 九、Docker 升级步骤

升级前建议备份 MySQL 数据卷或数据库。

```bash
cd /Users/xhr_mac/server/iot_control_platform
git fetch --tags
git checkout v0.10.0
docker compose up -d --build
```

容器启动时 backend 会自动执行 migrate。启动完成后：

```bash
docker exec iot-backend python manage.py showmigrations projects automation
docker exec iot-backend python manage.py sync_plugins --prune
docker ps
docker logs iot-backend --tail 50
docker logs iot-mqtt-runner --tail 50
```

验证前端产物：

```bash
docker exec iot-frontend sh -c "grep -o 'index-[A-Za-z0-9_-]*\.js' /usr/share/nginx/html/index.html"
```

## 十一、已知边界

- 调度器为单线程顺序执行，慢命令可能延后后续规则和控制方案。
- Project 的冗余 `project` / `section` 外键一致性主要由 serializer 保证。
- P&ID 当前主要由 InstrumentNode 绑定传感器；泵、阀等设备图元的命令交互仍是扩展项。
- 仿真 Web UI 是独立开发工具，不属于生产 Docker Compose 服务。
