# 0.9 版本更新日志

## 变更概览

本次更新围绕**自动化规则系统全面升级**展开，从脚本执行引擎、传感器/设备包装 API、前端编辑体验三个层面系统性改进，同时清理项目中散落的示例文件，统一归入 docs 文档库。

主要改动：

- **脚本执行引擎**：支持无类的顶层 `loop()` 函数风格，简单规则无需再套类
- **SensorWrapper 扩展**：新增 `.history()` / `.average()` / `.is_online` / `.refresh()` 方法，解锁时间窗口类自动化逻辑
- **DeviceWrapper 扩展**：同步新增 `.is_online` / `.refresh()`，状态惰性加载
- **前端编辑器**：textarea → CodeMirror（Python 语法高亮 + One Dark 主题）
- **设备选择器**：文本框 → 可搜索下拉选择器，选中后自动展示字段/命令提示
- **模板自动生成**：「插入模板」按当前 device_list 自动生成含变量名和注释的脚本骨架
- **文档清理**：自动化示例现统一维护在 `AutomationRules_design.md`，P&ID 图元扩展说明维护在 `project_guide.md`。

## 一、脚本执行引擎改进

### 支持顶层 `loop()` 函数

之前引擎只识别带 `loop()` 方法的类。现在同时支持直接定义顶层 `loop()` 函数，适合不需要辅助方法的简单规则：

```python
from engine import sensors, devices

def loop() -> bool:
    sensor = sensors.get('DHT11-001')
    if not sensor or not sensor.is_online:
        return False
    temp = sensor.current_state.get('temperature', 0)
    if float(temp) > 30:
        devices.get('fan_001').send_command('turn_on', {})
        return True
    return False
```

引擎查找顺序：先找带 `loop()` 方法的类，找不到再找顶层 `loop()` 函数。

### 移除 `__init__` = "setup()" 的误导说法

原设计文档和示例脚本把 `__init__` 类比为 Arduino 的 `setup()`，但引擎每次调用都会重新实例化 controller，`__init__` 并不会跨周期保留状态，这一比喻是误导性的。

现在推荐直接在 `loop()` 内调用 `sensors.get()` / `devices.get()`，与 Arduino 的 `loop()` 读引脚行为一致。

## 二、SensorWrapper / DeviceWrapper API 扩展

### 新方法一览

| 方法 / 属性 | 说明 |
|---|---|
| `sensor.current_state` | 最新 SensorData.data（首次访问时查 DB，本次 loop 内缓存） |
| `sensor.refresh()` | 强制重新从 DB 读取，刷新缓存 |
| `sensor.history('field', n=10)` | 最近 n 条字段值，时间升序，最新在最后 |
| `sensor.average('field', minutes=5)` | 最近 N 分钟均值（float），无数据返回 None |
| `sensor.is_online` | 是否在线：3 分钟内有数据上报 |
| `device.refresh()` | 同上，针对设备状态 |
| `device.is_online` | 3 分钟内有数据上报 |

### 实现要点

**惰性加载**：`current_state` 原来在 `build_sensors()` 阶段（每次 `execute_rule` 开头）预先查询并缓存到匿名属性。现在改为首次访问时惰性查询，避免"传感器注册了但 loop 里没用到"时的无效查询。同一个 `loop()` 执行内多次访问 `current_state` 读同一缓存，结果一致；需要最新值时调 `refresh()`。

**SensorWrapper / DeviceWrapper 从匿名类改为真实类**：原来用 `type('SensorWrapper', (), {...})()` 动态创建，无法添加方法。现在是正式的 Python class，可继承、可扩展。

### 使用示例

```python
from engine import sensors, devices

class TrendAlert:
    def loop(self) -> bool:
        sensor = sensors.get('DHT11-001')
        if not sensor or not sensor.is_online:
            return False

        # 过去 5 分钟平均温度
        avg = sensor.average('temperature', minutes=5)
        if avg is None:
            return False

        # 最近 10 条历史值（判断趋势）
        history = sensor.history('temperature', n=10)
        if len(history) >= 3 and history[-1] > history[-3] + 2:
            # 温度在上升
            devices.get('fan_001').send_command('turn_on', {})
            return True
        return False
```

## 三、前端编辑器升级

### 代码编辑器：textarea → CodeMirror

`AutomationDetailView.vue` 的脚本编辑区从裸 `<textarea>` 升级为 CodeMirror 6，带来：

- Python 语法高亮（关键字、字符串、注释着色）
- One Dark 主题（与终端区颜色风格统一）
- 括号高亮匹配、选区显示

安装的包：
```
vue-codemirror@6.1.1
@codemirror/lang-python@6.2.1
@codemirror/theme-one-dark@6.1.3
```

### 设备选择器：文本框 → 可搜索 Select

关联设备/传感器面板的 `device_id` 列从纯文本输入框改为可搜索下拉选择器：

- 选项列表来自新增的后端接口 `GET /api/automation-rules/available-sources/`，返回所有传感器（含 `data_fields`）和设备（含 `commands`）
- 选项显示 ID + 名称，防止拼错
- 仍支持 `allow-create` 手动输入（兼容非标准 ID）

选中设备后，行下方自动展示字段/命令提示：

```
[字段]  temperature  ·  humidity
[命令]  turn_on  ·  turn_off  ·  set_speed
```

### 「插入模板」自动生成

之前插入的是通用骨架，`SENSOR_ID = ''` 需要手动填。现在根据当前 device_list 自动生成：

- 按 device_id 推导 Python 变量名（替换特殊字符为 `_`）
- Sensor 自动生成 `sensors.get(...)` 行 + `current_state` 读取示例
- Device 自动生成 `devices.get(...)` 行 + `send_command` 注释
- 空 device_list 时仍生成通用骨架

### 脚本编写指南更新

指南面板移除了 `__init__` / setup() 说法，补充了新 API 文档，加入了带 `average()` 的完整示例代码块。

  "sensors": [

## 六、已知限制 / 后续可优化

- **调度器单线程**：scheduler 在同一线程内顺序执行所有规则；若某条规则的 `send_command` 耗时（3s 超时 × N 台设备），会推迟后续规则的执行。后续可考虑每条规则跑独立线程/协程
- **`average()` 不带索引**：`SensorData` 按 `timestamp` 查询没有复合索引（`sensor + timestamp`），数据量大时这个查询会扫表。后续可在 migration 里加 `db_index=True` 覆盖
- **CodeMirror 代码块体积**：`AutomationDetailView` 打包后约 484kB（gzip 163kB），是目前单页最大的 chunk。如需优化可改为动态 import
- **设备选择器的 `allow-create`**：允许手输不在列表里的 ID，合法性验证在后端执行时才暴露（找不到传感器时 get() 返回 None）；前端暂不做额外校验
