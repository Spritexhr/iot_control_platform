# 0.8 版本更新日志

## 变更概览

本次更新主线是**全面实时化**——把之前只覆盖 EB 大屏的 SSE 单点推送，统一升级成基于 **Django Channels + Redis** 的 WebSocket 通用实时层，覆盖全平台所有"需要不刷新就能看到变化"的场景；同时把 MQTT 客户端从 web worker 中拆出独立进程，修复了多 worker 部署下每条 MQTT 消息被处理 N 次的隐藏 bug。

主要改动：

- **实时层重构**：Django Channels + channels_redis + JWT 中间件；6 类 consumer（传感器/设备/自动化/MQTT 系统/插件/联调 ping）按"资源 × 资源ID"划分 group
- **5 个页面去轮询**：自动化列表/详情（原 5s 轮询）、传感器/设备详情（原命令后 1s setTimeout）、AppHeader MQTT 状态（原 15s 轮询）全部改 WS；传感器/设备**列表页**和 Dashboard 之前**完全没有**自动刷新，本次也接入 WS
- **EB 大屏从 SSE 迁移到 WS**：删旧 `Broadcaster` 类、删 SSE 端点；EBPlantView / PlantDiagramView 切换到 `useWebSocket`，体验不变（多 tab 同步、自动重连均保留）
- **MQTT 客户端进程拆分**：新增 `mqtt_runner` management command 与 docker 独立 service；backend worker 切到"仅发布"模式（不订阅 topic，但保留命令下发能力），消息只在一个进程落库
- **部署切到 ASGI**：`gunicorn -k uvicorn.workers.UvicornWorker` 同端口承载 HTTP + WS；docker-compose 新增 `redis` service + `mqtt_runner` service；nginx 加 `/ws/` 反代
- **前端通用 composable**：`useWebSocket` 与原 `useSSE` API 兼容，支持 25s 心跳、4001 → refresh token → 重连、指数退避

---

## 一、Channels + Redis + ASGI 基建

### 背景

0.7 之前的实时推送是 EB 大屏专用的 SSE 端点（`/api/plugins/eb_plant/stream`），基于 `services/realtime/latest_values.py` 的进程内 `Broadcaster`（`queue.Queue` + 订阅列表）。三个根本限制：

1. **进程内单点**：多 worker 部署时只有"接到 SSE 请求的那个 worker"会推送，其他 worker 收到 MQTT 消息后广播的 `Broadcaster` 是另一个进程的副本
2. **只服务一个插件**：通用层不能复用，每加一个需要实时性的页面都得复制一遍
3. **EventSource 单向**：客户端没法主动通知后端（如心跳、ack）

### 变更

#### 后端依赖与配置

`requirements.txt` 新增：

```
channels>=4.1
channels-redis>=4.2
daphne>=4.1
uvicorn[standard]>=0.30
redis>=5.0
```

`config/settings.py`：

- `INSTALLED_APPS` 顶部加 `"daphne"`（Channels 文档明确要求 daphne 在 staticfiles 前），后面加 `"channels"`，本地应用加 `"services.realtime.apps.RealtimeConfig"`
- 末尾新增：

  ```python
  ASGI_APPLICATION = "config.asgi.application"
  _REDIS_URL = os.environ.get("REDIS_URL", "redis://127.0.0.1:6379/0")
  CHANNEL_LAYERS = {
      "default": {
          "BACKEND": "channels_redis.core.RedisChannelLayer",
          "CONFIG": {"hosts": [_REDIS_URL], "capacity": 1500, "expiry": 30},
      }
  }
  ```

#### `config/asgi.py` 完整改写

从原来的裸 `get_asgi_application()` 改为 `ProtocolTypeRouter`：

```python
django_asgi_app = get_asgi_application()  # 必须先初始化再 import channels
# ...
application = ProtocolTypeRouter({
    "http": django_asgi_app,
    "websocket": JwtAuthMiddleware(URLRouter(websocket_urlpatterns)),
})
```

启动时遍历 `discover_plugins()` 把启用插件的 `routing.py` 动态合入 `websocket_urlpatterns`（参考下面"插件 WS 自动挂载"）。

#### 新增 `services/realtime/` 子模块

按 AppConfig 注册为独立 Django app（label `services_realtime`）：

```
iot_control_platform/services/realtime/
├── __init__.py
├── apps.py          # RealtimeConfig（ready() 内 import signals）
├── dispatch.py      # group 命名 + publish_*() 同步桥
├── signals.py       # 主模型 post_save handler（transaction.on_commit）
├── consumers.py     # 6 个主层 Consumer + _BaseAuthedConsumer
├── middleware.py    # JwtAuthMiddleware（query string token）
└── latest_values.py # 平移自原位置；删 Broadcaster 类
```

#### Group 命名规则

按"**资源 × 资源ID**"组织（不按"插件 × 事件"）：

| Group 名 | 用途 |
|---|---|
| `sensors.{sensor_id}` / `sensors.all` | 单传感器订阅 / 全量 |
| `devices.{device_id}` / `devices.all` | 单设备订阅 / 全量 |
| `automation.rules` | 自动化规则状态变更 |
| `system.mqtt` | MQTT broker 连接状态 |
| `plugins.{plugin_code}` | 插件自定义流（如 EB 大屏） |

理由：插件×事件命名会让通用 consumer 跟具体插件耦合，新增插件就要改主层；按资源命名让"传感器详情页"和"传感器列表页"自然落到同一 group，插件如需自定义另开 `plugins.{code}` 即可。

#### `services/__init__.py` 清空

原来在包顶层 import `mqtt_service` 等，Django apps populate 阶段 import `services.realtime` 时会拉链触发 models 的 import → `AppRegistryNotReady`。现改为纯命名空间包，按需 `from services.<sub> import ...` 即可。

#### WS 鉴权（`services/realtime/middleware.py`）

浏览器 WebSocket 不支持自定义 Authorization header，握手时只能通过 query string 带 token：

```
ws://host/ws/sensors/<id>/?token=<jwt>
```

`JwtAuthMiddleware` 用 `rest_framework_simplejwt.tokens.AccessToken` 校验，注入 `scope["user"]`；token 无效返回 4001 close code，前端 `useWebSocket` 收到后自动跑一次 refresh → 重连。

| 文件 | 变更 |
|---|---|
| `requirements.txt` | 新增 5 个依赖 |
| `config/settings.py` | INSTALLED_APPS / ASGI_APPLICATION / CHANNEL_LAYERS |
| `config/asgi.py` | ProtocolTypeRouter + 插件动态挂载 |
| `services/__init__.py` | 清空顶层 import（修复 AppRegistryNotReady） |
| `services/realtime/{apps,dispatch,signals,consumers,middleware}.py` | 全新建 |
| `services/realtime/latest_values.py` | 删 `Broadcaster` 类；`ingest_sensor_data` 出口换为 `dispatch.publish_plugin_sample` |
| `services/realtime/__init__.py` | 删 `broadcaster` 导出 |

## 二、传感器实时推送（M2）

### 变更

`services/realtime/signals.py` 加两个 receiver：

- `SensorData.post_save` → `dispatch.publish_sensor_data(sensor_id, payload)` → 广播 `sensors.{id}` + `sensors.all`

Consumer：`SensorListConsumer`（`/ws/sensors/`）+ `SensorStreamConsumer`（`/ws/sensors/<sensor_id>/`），都同时 handle data + status 两类事件。

## 四、自动化规则实时推送（M4）

### 背景

`AutomationView.vue` 和 `AutomationDetailView.vue` 各自有一个 5s `setInterval` 拉规则状态。规则启动/停止/异常感知最差 5s 延迟，是全平台实时性最差的痛点。

## 六、MQTT 客户端进程拆分（M5 后半）

### 背景

`gunicorn --workers 2` 部署下，每个 worker 在 `AppConfig.ready()` 都会独立起一个 paho MQTT 客户端，订阅同一组 topic。结果是**每条 MQTT 消息进来被两个 worker 各处理一遍**——SensorData 写库两次（被 UniqueConstraint 兜底但浪费）、signal 触发两次、dispatch 推两次、前端 WS 收到重复推送。这是 0.7 之前就存在的隐藏 bug，0.8 升 WS 后症状更明显。

### 变更

#### MQTT 客户端两种运行模式（`sensors/apps.py`）

| 模式 | 触发 | 行为 |
|---|---|---|
| 完整模式 | `mqtt_runner` 命令 或本地开发单进程 | 连接 + 绑定 publisher + **订阅 topic + 注册 handler** |
| 仅发布模式 | 环境变量 `IOT_MQTT_RUNNER=false` | 连接 + 绑定 publisher，**不订阅** |
| 关 | `test` / `migrate` / `makemigrations` | 不启动 |

判定函数：

```python
def _is_publisher_only(self) -> bool:
    if 'mqtt_runner' in sys.argv:
        return False
    return os.environ.get('IOT_MQTT_RUNNER', '').lower() in ('false', '0', 'no')

def _start_mqtt_service(self, subscribe: bool = True) -> None:
    # ...
    if not mqtt_service.connect(timeout=5): return
    sensor_command_send_service.set_mqtt_service(mqtt_service)
    device_command_send_service.set_mqtt_service(mqtt_service)
    if subscribe:
        mqtt_service.setup_sensor_data_handler()
        mqtt_service.setup_sensor_status_handler()
        mqtt_service.setup_device_status_handler()
```

#### 新增独立 MQTT 进程

- 新建 `sensors/management/commands/mqtt_runner.py`：sensors AppConfig.ready() 在 `mqtt_runner in sys.argv` 时启动完整模式 MQTT，本命令主线程阻塞 + 监听 SIGTERM 优雅退出
- `docker-compose.yml`：
  - `backend` service 加 `IOT_MQTT_RUNNER=false` env → 仅发布模式
  - **新增** `mqtt_runner` service，同镜像，command 跑 `python manage.py mqtt_runner`，`depends_on: backend` 等首次 migrate 完成

#### 最终架构

| 容器 | MQTT 行为 | 数量 |
|---|---|---|
| `iot-backend` | 连接 + 发布命令；**不订阅** | 2 worker |
| `iot-mqtt-runner` | 连接 + 订阅 + handler → 写库 → signal → channel layer | 1 |

3 个 MQTT 客户端连同一个 broker（每个 client_id 含 pid 不冲突），但**只有 mqtt_runner 处理上报消息**——既解决重复处理，又保留 backend 命令下发能力。

#### 副作用：`system.mqtt` 多次推送

3 个进程的 `_on_connect` 都会触发 `_publish_system_status`，前端订阅 `/ws/system/mqtt/` 在 broker 重连时会收到 3 条相同状态消息。UI patch 同样 ref 不抖动，**当前接受**。如未来要消除冗余，让 publisher 模式不调 `_publish_system_status` 即可。

## 七、前端通用 useWebSocket composable

### 新建 `frontend/src/composables/useWebSocket.js`

API 兼容 0.7 的 `useSSE`：`{ status, displayStatus, start, stop }`，多一个 `send()`。要点：

- **URL 注入函数式**：`buildUrl()` 每次重连前调一次，便于动态拼装（如详情页 sensor_id 异步加载后再 `start()`）
- **token 自动注入**：从 `localStorage['iot-access-token']` 取，附到 query string
- **4001 → refresh → 重连**：连续两次 4001 置 `unauthorized`，由调用方决定跳登录
- **25s 心跳**：客户端 `{action: "ping"}`，后端 consumer 在 `_BaseAuthedConsumer.receive_json` 回 `{action: "pong"}`
- **指数退避重连**：base 1s、上限 30s；`manualClosed` 标志避免 `stop()` 后还重连
- **防抖状态**：connecting/error 持续超过 `transientMs`（默认 2s）才刷 UI，避免浏览器瞬时抖动
- **`onScopeDispose(stop)` 自清理**：必须在 Vue setup 顶层调用，详情页等需异步拿 ID 的场景用 `autoStart: false`

工具：`buildWsUrl(path)` 根据 `location.protocol` 推 `ws://` 或 `wss://`。

### nginx + Vite proxy

`frontend/nginx.conf` 加 WebSocket 反代：

```nginx
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}

server {
    # ...
    location /ws/ {
        proxy_pass http://backend:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_buffering off;
        proxy_read_timeout 24h;   # 长连接
        # ...
    }
}
```

**同时删除**了 0.7 时为 SSE 写的 `location ~ ^/api/plugins/[^/]+/stream$` 块（M3 后已是死代码）。

`frontend/vite.config.js` 加 `/ws` 代理（`ws: true`）给 `npm run dev` 用。

### 注意事项

- **首次启动需要 Redis 在跑**：channels_redis 在 group_send 时连不上会失败（dispatch 有 try/except 兜底不崩，但 WS 推送链路完全失效）
- **JWT 过期阈值**：access token 默认 2h，WS 连接超过 2h 会收到 4001；`useWebSocket` 已自动处理 refresh + 重连
- **EB 大屏旧 SSE 端点 `/api/plugins/eb_plant/stream` 已删**：前端老缓存可能还在请求，刷新页面即可
- **`SECRET_KEY` 没变**：JWT 鉴权对 WS 与 HTTP 同样使用同一 SECRET_KEY

## 九、设计要点速查

1. **Group 命名按"资源 × 资源ID"**：主层 consumer 与具体插件解耦；插件自建 `plugins.{code}` group
2. **双层广播**：主层广播主模型原始数据；插件层（如 EB）保留 binding 过滤 + 阈值计算，避免主层反向依赖插件
3. **`transaction.on_commit` 延迟广播**：保证事务回滚不推消息、也不阻塞 INSERT 事务
4. **MQTT 进程拆分**：subscriber 唯一（mqtt_runner），publisher 多份（backend worker）；二者通过 redis 共享 channel layer
5. **WS 鉴权用 query string token**：浏览器 WebSocket 不支持自定义 header；4001 close code 触发前端 refresh + 重连
6. **view-level 订阅，不建 store**：每页 setup 顶层 `useWebSocket`，`onScopeDispose` 自动断连；除非确实需要跨页面共享 state（如 EB 大屏的 `plantStore`）
7. **`useWebSocket` API 与 `useSSE` 兼容**：迁移成本最小，相同的 `{status, displayStatus, start, stop}` 接口

## 十、已知限制 / 后续可优化

- **`system.mqtt` 重复推送 3 次**：backend 2 worker + mqtt_runner 共 3 个 MQTT 客户端各推一次连接状态。UI 不抖但有冗余，可在 publisher 模式禁推系统状态消除
- **Dashboard 顶部计数走增量算**：基于 `recent_sensors` / `recent_devices` 顶部 N 行的 status 翻转累加；如果 N 外的传感器上下线，顶部 `X 在线` 计数会偏，刷新按钮可矫正。彻底方案是后端推一个 `system.counts` 事件
- **`AppConfig.ready` 期间 DB 警告**：daphne worker 启动时 `configure --init` 会触发 `Accessing the database during app initialization is discouraged` 警告，功能正常，是 Django 6 新加的提示
- **日志轮转 FileNotFoundError**：`config/logging_config.py` 自定义 `doRollover` 删旧文件时不存在会抛错（被 logging 内部捕获，不影响业务），需要包一层 `try/except FileNotFoundError`
- **WS 没有按资源做权限校验**：当前只验"已登录"，任何登录用户都能订阅任意 `sensors/{id}`。如有租户隔离需求，需要在 consumer `connect` 里加 `IsAdminUser` 或资源 ACL 检查
- **`useWebSocket` 没有可见性感知**：浏览器标签页切到后台仍保持连接 + 心跳。如要节省资源，可监听 `document.visibilitychange` 在隐藏时 `stop()`、显示时 `start()`
- **`mqtt_runner` 单进程**：若 broker 消息量极大，会成为单点瓶颈。可考虑拆 topic 到多个 runner（按 sensor_id 哈希分片）
- **i18n 普及未推进**：0.7 提到的 ~345 处硬编码中文未在本版本处理，留到后续
