# 乙苯装置 IoT 辅助监测预警系统 - 实施方案

> **项目背景**:化工设计竞赛差异化亮点设计
> **工艺对象**:乙苯(EB)生产装置(基于 Luyben 2011 AIChE Journal 论文)
> **系统定位**:独立于 DCS 主控系统的智能辅助监测与预警层
> **文档版本**:v2.0(2026-05-13 基于现有 IoT 控制平台重构)
> **预计工期**:2 个月(8 周,含缓冲)

---

## 修订说明(v1.0 → v2.0)

v1.0 是"全独立栈方案"(Mosquitto + Node-RED + InfluxDB + Grafana),把所有组件都拼现成工具,
6-8 天可以跑通,但本质上是个"搭积木 demo",和市面上同类作品没有差异。

v2.0 改为**"以自研 IoT 平台为主体"的整合方案**。原因:

1. 项目仓库已有完整的 Django + Vue3 + MQTT 控制平台、设备/传感器模型、虚拟设备模拟器、
   中英文切换、Claude 风格 UI 等基础设施,这是参赛队伍中**独有的硬资产**。
2. 答辩故事从"我用 Grafana 拼了个监控面板"升级为"我自研了一套面向化工场景的工业软件,
   并以乙苯装置为示范",评委记忆点和加分维度完全不同。
3. 工期从 6-8 天放宽到 8 周,有空间做出 P&ID 动态可视化、可视化规则引擎等真正的差异化亮点。

## 一、项目定位

### 1.1 核心定位

本系统是一套**独立于安全控制系统(SIS/DCS)的智能预警与辅助监测层**,简称**辅助监测层**。

### 1.2 关键边界(必须明确)

- ❌ **不参与**任何闭环控制
- ❌ **不进入**安全联锁回路(SIS)
- ❌ **不替代**DCS/PLC 的主控功能
- ✅ **只做**旁路监测、预警通知、数据沉淀
- ✅ **符合**IEC 62443 工业信息安全标准对监测系统的定位
- ✅ 关键控制仍由成熟的 DCS/PLC 系统完成

### 1.3 设计哲学

> 工业 4.0 中的 "IT/OT 融合" 理念:不打破现有控制系统的可靠性,
> 通过低成本 IoT 技术 + **自研工业软件平台**补充传统监测的盲区。

## 二、技术架构

### 2.1 与原方案的对比

| 维度 | v1.0(独立栈) | v2.0(平台整合) |

| 数据生成 | 独立 Python 脚本 | 复用平台 `simulation` 模块,统一配置 |

### 2.4 明确排除的工具

- ❌ **Node-RED**:逻辑写在 Django 里更可控,也避免架构碎片化
- ❌ **InfluxDB**:Timescale 已能覆盖,少一个服务好运维
- ❌ **Grafana**(默认不用):前端能力够,且需要保持 Claude 风格统一;
  如果工期紧张,作为 plan-B 嵌入 iframe 也行,但不是首选
- ❌ Aspen Dynamics(动态仿真过于复杂)
- ❌ Factory I/O(3D 太重,投入产出比低)
- ❌ OPC 通信(不对接真实 DCS)
- ❌ PLC 编程(明确边界,不做控制)

#### 关键流量

|------|---------|--------|------|
| FT-101 | 乙烯进料流量 | 630.6 kmol/h | 主进料 |
| FT-102 | 苯进料流量 | 630.6 kmol/h | 主进料 |

**合计约 20 个监测点**,覆盖全装置关键参数。

### 3.2 在平台中的落地方式

所有监测点都通过现有的"传感器"模型注册,不新建数据结构:

- 利用已有的 `Device` + `Sensor` 模型,新增装置 `EB-Plant`,挂载 ~20 个 sensor
- 每个 sensor 通过 `mqtt_topic` 字段绑定到 `plant/{area}/{tag}`
- 在 sensor 元数据里扩展字段:`unit`、`normal_value`、`hi_threshold`、`lo_threshold`、`severity`
- simulation 模块按这套配置发数,规则引擎按这套配置判定

### 3.3 重点监测对象:DEB 循环流量

论文中的"雪球效应"是本系统的**核心价值场景**:

- 正常值:251~282 kmol/h
- 进料杂质扰动后:可在 30 小时内爬升至 600 kmol/h
- 传统阈值报警:等到超阈值时已经偏离严重
- **本系统优势**:趋势预警可提前 5-10 小时发现异常

## 四、核心亮点设计

### 亮点 1:DEB 循环流量趋势预警(杀手锏)

**原理**:滑动窗口 + 线性回归,斜率超阈值即预警。

**实现位置**:Django 规则引擎中的 `TrendRule` 类型

**核心算法**(Python,运行在规则引擎 worker):

```python
class TrendRule(BaseRule):
    sensor: Sensor          # 监测点(FT-401)
    window_seconds: int      # 1 小时
    slope_threshold: float   # kmol/h per hour

    def evaluate(self, ts_buffer):
        points = ts_buffer.window(self.sensor, self.window_seconds)
        if len(points) < 10:
            return None
        slope = linear_regression(points).slope
        if slope > self.slope_threshold:
            return Alert(
                level='WARNING',
                message=f'DEB 循环流量异常爬升,斜率 {slope:.2f} kmol/h²',
                predicted_exceed_time=estimate_exceed(points, slope),
            )
```

**演示效果**:对比"传统阈值报警 vs 趋势预警",证明可提前 X 小时发现问题。

### 亮点 2:多参数关联报警

单参数报警容易误报,组合判断更靠谱:

| 规则 | 触发条件 | 推断异常 | 严重程度 |
|------|----------|----------|----------|
| R1 冷却异常 | TT-101 上升 + FT-101 正常 + FT-501 下降 | 反应器冷却系统失效 | 高 |
| C1 堵塔预警 | TT-301 下降 + 再沸器蒸汽上升 | 塔效率下降,可能堵塔 | 中 |
| 原料污染 | FT-401(DEB) 异常爬升 + 苯进料质量异常 | 进料含杂质 | 中 |
| 真空失效 | PT-301 或 PT-401 上升 + 冷凝器温度上升 | 真空机组故障 | 高 |
| 蒸汽包危险 | LT-102 持续下降 + TT-101 上升 | 蒸汽包干烧风险 | **严重** |

**实现位置**:规则引擎中的 `CompositeRule`,支持 AND/OR 组合 + 持续时间窗口。

### 亮点 3:P&ID 动态可视化大屏(评委记忆点)

> 这是 v2.0 相对原方案最大的增量亮点,也是最能体现自研工业软件价值的部分。

**做什么**:在 Vue3 前端新建一个全屏页面,渲染**简化版乙苯装置 P&ID**(SVG),
所有传感器位置实时跟随 MQTT 数据动起来。

**动态元素**:
- 罐体/塔器液位:`<rect>` 的高度按 LT 值线性变化
- 管道流向:`<path>` 用 `stroke-dasharray` + `animation` 做"流动"效果,流速 ∝ 流量
- 温度热力:管段颜色按温度做渐变(蓝→绿→黄→红)
- 报警闪烁:命中规则的位置外圈红色 `animation: blink`
- 数值气泡:每个仪表位置浮一个数值标签,鼠标 hover 出趋势小图

**为什么这是亮点**:
- Grafana 做不出来,Node-RED Dashboard 做出来很丑
- 评委一眼能看懂"这就是化工厂在跑",远比时序曲线直观
- 体现了"化工 + 软件"复合能力,这是化工设计竞赛真正稀缺的

**技术实现**:
- SVG 手绘 P&ID 骨架(可以从 AutoCAD 导出再简化)
- Vue3 组件按 sensor 配置生成动态层
- 数据通过 WebSocket(Django Channels)推送,1Hz 刷新
- 用 Pinia 集中管理实时点位状态

### 亮点 4:可视化规则配置中心("无代码组态报警")

**做什么**:在 Vue3 里做一个规则编辑页面,普通工程师不用写代码就能配置:
- 选传感器
- 选规则类型(阈值 / 趋势 / 关联)
- 填阈值或勾选关联规则的组合条件
- 实时生效(无需重启 worker)

**为什么这是亮点**:
- 答辩故事直接升级为"自研工业组态软件"
- 评委容易理解:把传统 PLC 工程师做的事用现代 Web 思路重做
- 工作量可控:本质是一套带校验的表单 + 序列化为 JSON 存数据库

### 亮点 5:移动端推送

**链路**:规则引擎 → 企业微信 webhook / Server酱 → 手机

**演示要点**:答辩现场点击"注入扰动"按钮,几秒后手机响,弹出报警消息。

### 亮点 6:历史数据回溯

**TimescaleDB + Vue3 时间轴回放**:
- 30 天历史数据
- Vue3 大屏支持"时间轴拖动",P&ID 可以倒回历史时刻"重演事故"
- 这比 Grafana 的时间选择器更有"故事性",答辩演示效果好

### 亮点 7:边缘智能(可选)

ESP32(Wokwi)端实现轻量级处理:
- 滑动平均滤波(降噪)
- 阈值预判(只在异常时上传)
- 数据压缩(变化超过 X% 才传)
- 演示"边缘-云协同架构"概念

工期紧时可砍,优先级最低。

### 7.1 故事线

**问题导入(30 秒)**

> 传统化工厂 DCS 系统监测点固定、报警阈值僵化、远程监控能力弱、数据沉淀少。
> 论文中提到的"雪球效应"扰动,传统系统等到报警时已偏离严重。

**方案提出(60 秒)**

> 我们**自研了一套面向化工场景的 IoT 辅助监测平台**,以乙苯装置为示范应用。
> 平台独立于 DCS 主控,通过 ESP32 + MQTT 架构实现全厂感知、智能预警、移动监控、数据沉淀。
> **强调**:不替代 DCS,不参与闭环控制。

**技术架构(60 秒)**

> 前后端全自研:Django + Vue3 + TimescaleDB + MQTT;
> 后端含自研规则引擎(阈值/趋势/关联三类规则);
> 前端含 P&ID 动态大屏 + 可视化规则配置中心;
> 与主流"拼装积木方案"(Node-RED + Grafana)相比,更贴合工业场景需要。

**实证演示(120 秒)**

> [现场点击"注入 DEB 杂质扰动"按钮]
> 观察:P&ID 大屏 FT-401 数值缓慢上升,管线颜色变化 → 趋势预警触发 →
> 报警事件流弹出红色卡片 → 手机收到企业微信报警 → 时间轴回放展示完整过程
> **关键对比**:传统阈值报警延迟 X 小时,我们的趋势预警提前 X 小时。

**边界声明(30 秒)**

> 本系统定位为辅助监测,符合 IEC 62443 安全标准。
> 关键控制由 DCS 完成,IoT 不参与安全联锁。这是工业 4.0 中"IT/OT 融合"的标准做法。

**收尾(15 秒)**

> 低成本、易扩展、智能化、**全栈自研**,为传统化工装置的数字化转型提供可行路径。

### 7.2 预期评委提问与回答

**Q1:为什么不直接用 DCS 做这些?**
A:DCS 报警阈值是固定的,趋势分析能力弱,且扩展监测点成本高(每点数千元)。
IoT 节点单点成本几十元,可低成本部署上百个监测点,做更精细的感知。两者是补充关系。

**Q2:为什么不用 Grafana / Node-RED 现成方案?**
A:现成方案能快速出 demo,但有三个局限:风格割裂、扩展受限、不能体现工程能力。
我们的目标不是"做出一个监控页面",而是"做出一个工业软件产品",所以选择自研。
P&ID 动态可视化、可视化规则配置这两个功能,在 Grafana 体系里很难做。

**Q3:MQTT 在工业现场可靠吗?**
A:正因为不可靠(相对工业总线而言),所以**只做辅助监测,不进控制回路**。
这正是我们方案的设计边界。

**Q4:网络安全怎么保证?**
A:物理隔离(IoT 网络与控制网络分开)、单向数据流(只读不写)、MQTT TLS 加密、
符合 IEC 62443 区域划分原则。

**Q5:有没有实际工业部署案例?**
A:中石化、中石油、巴斯夫等都有类似的"辅助监测云平台"项目,本质相同。
我们的创新点在于**用低成本 IoT 硬件 + 自研工业软件平台 + 趋势预警算法**实现了这类系统。

**Q6:这套软件能复用到其他装置吗?**
A:可以。装置、传感器、规则全部数据驱动,接入新装置只需要:
①注册传感器并绑定 MQTT topic;②画一张新的 P&ID SVG;③配置规则。
无需改代码。
